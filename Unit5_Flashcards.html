Organizing Class Design
Q: Steps Involved in Organizing Class Design
  • Purpose: Transform initial analysis model into a well-structured design model
  • 1) Identify Design Classes
  • Convert analysis classes into design classes (Entity, Control, Boundary)
  • 2) Define Class Responsibilities
  • Assign clear responsibilities, follow Single Responsibility Principle
  • 3) Identify Attributes
  • Determine data required by each class, decide data types and visibility
  • 4) Identify Operations (Methods)
  • Define operations needed to fulfill responsibilities, derived from use cases
  • 5) Establish Relationships
  • Association, Aggregation/Composition, Inheritance, Dependency
  • 6) Adjust Inheritance
  • Validate IS-A relationships, introduce abstract classes, replace inheritance with composition
  • 7) Apply Design Patterns
  • Use patterns like Factory, Strategy, Observer
  • 8) Optimize Class Design
  • Reduce coupling, increase cohesion, refine access control
Q: Bridging Gap from High-Level Requirements to Low-Level Services
  • Concept: High-level requirements ‚Üí What system should do; Low-level services ‚Üí How system does it
  • Step 1: Analyze High-Level Requirements - Study use cases, understand system behavior
  • Step 2: Identify Conceptual Classes - Identify domain objects (Account, Transaction, ATM)
  • Step 3: Convert to Design Classes - Add software-specific responsibilities (ATMController, TransactionManager)
  • Step 4: Define Attributes and Operations - Convert requirements into attributes and methods (withdrawCash(), validatePIN())
  • Step 5: Assign Responsibilities - Decide which class performs which service
  • Step 6: Define Interactions - Decide how classes collaborate
  • Step 7: Apply Design Principles & Patterns - Use patterns to manage complexity
  • Step 8: Refine into Low-Level Services - Finalize method signatures
Adjustment of Inheritance
Q: How Adjustment of Inheritance is Done in Class Design?
  • Definition: Process of modifying, refining, or restructuring inheritance hierarchies to ensure correct abstraction, reuse, and minimal coupling
  • Why Adjustment is Needed:
  • Analysis models are often conceptual
  • Direct inheritance may cause tight coupling
  • Improper inheritance may violate OO principles
  • Need to optimize design for implementation
Q: How Adjustment of Inheritance is Done (Steps)
  • 1) Identify Incorrect "IS-A" Relationships
  • Check whether subclass truly is a type of superclass
  • Example: Car IS-A Engine ‚ùå (Wrong ‚Üí use composition)
  • 2) Replace Inheritance with Composition
  • Prefer HAS-A over IS-A when behavior is reused but relationship is not IS-A
  • Example: Car HAS-A Engine
  • 3) Introduce Abstract Superclasses
  • Move common attributes and operations to abstract class
  • Example: Account as abstract superclass for SavingsAccount, CurrentAccount
  • 4) Push Down Responsibilities
  • Move behavior to subclasses if it is not common
  • 5) Pull Up Common Features
  • Identify duplicate attributes or methods in subclasses and move to superclass
  • 6) Reduce Depth of Inheritance Hierarchy
  • Avoid very deep inheritance trees
  • 7) Use Interfaces Instead of Inheritance
  • When multiple inheritance is required or to avoid tight coupling
Q: Importance of Adjustment of Inheritance
  • 1) Improves Design Quality
  • Ensures proper abstraction, eliminates wrong hierarchies
  • 2) Promotes Reusability
  • Common behavior is reused correctly, avoids code duplication
  • 3) Reduces Coupling
  • Prevents unnecessary dependencies, makes system easier to modify
  • 4) Improves Maintainability
  • Changes affect fewer classes, cleaner class hierarchy
  • 5) Supports Polymorphism
  • Ensures methods are overridden correctly, enables correct dynamic binding
  • 6) Prevents Fragile Base Class Problem
  • Avoids ripple effect of superclass changes
Q: Steps for Adjustment of Inheritance
  • Step 1: Review Class Hierarchy - Examine all inheritance relationships, validate IS-A relationship
  • Step 2: Identify Commonality and Variability - Common features ‚Üí superclass, Variable features ‚Üí subclasses
  • Step 3: Eliminate Redundant Attributes and Methods - Remove duplication across subclasses
  • Step 4: Replace Improper Inheritance - Use aggregation/composition where required
  • Step 5: Introduce Interfaces or Abstract Classes - For shared behavior without forced inheritance
  • Step 6: Optimize Inheritance Depth - Avoid unnecessary multilevel inheritance
Identifying Use Cases and Actors
Q: What is a Use Case Diagram?
  • Use Case Diagram shows: External entities (actors), System functionality (use cases), Interaction between actors and system
  • Purpose: Capture functional requirements from user's perspective
Q: How to Identify Actors?
  • Actor Definition: External entity that interacts with system to achieve a goal
  • 1) Identify Who Uses the System
  • Human users who interact directly (Customer, Admin, Operator)
  • 2) Identify External Systems
  • Other systems that exchange data (Bank Server, Payment Gateway)
  • 3) Identify Hardware or Devices
  • Devices interacting with system (Card Reader, Printer, Sensor)
  • 4) Identify Roles, Not Individuals
  • Actor represents a role (Customer, not "Rahul")
  • 5) Ask Key Questions:
  • ‚Ä¢ Who starts the system?
  • ‚Ä¢ Who uses the system's services?
  • ‚Ä¢ Who maintains or administers the system?
  • ‚Ä¢ Who gets information from the system?
Q: Actor Classification
  • Primary Actor: Initiates use case
  • Secondary Actor: Supports system
Q: How to Identify Use Cases?
  • Use Case Definition: Set of actions performed by system to produce observable result for an actor
  • 1) Study System Requirements
  • Functional requirements directly map to use cases
  • 2) Identify Actor Goals
  • Each goal of actor becomes a use case (Customer ‚Üí Withdraw Cash)
  • 3) Use Verb‚ÄìNoun Format
  • Use cases represent actions (Check Balance, Print Receipt)
  • 4) Identify System Services
  • Each service provided by system is a use case
  • 5) Avoid UI-Level Actions
  • Focus on what system does, not how (‚úì Submit request, ‚úó Click button)
  • 6) Check Completeness
  • Each actor should be linked to at least one use case
Q: ATM System Example
  • Identified Actors: Customer (Primary), Bank Server (Secondary), Maintenance Staff
  • Identified Use Cases: Insert Card, Validate PIN, Withdraw Cash, Check Balance, Print Receipt

6) Optimization of Object Communication Reduce unnecessary message passing Improved performance, reduced complexity

8) Optimization of Data Storage Use efficient data structures, caching, pooling Better performance, reduced memory

Q: Importance of Design Optimization
  • Improves system quality
  • Reduces future maintenance cost
  • Enhances performance
  • Ensures design scalability
Finding and Defining Classes
Q: What is a Class?
  • Definition: A blueprint that defines the attributes (data) and operations (behavior) of objects
Q: Guidelines for Finding Classes
  • 1) Noun Phrase Identification
  • Study requirements, identify nouns (ATM, Account, Card, Transaction)
  • 2) Identify Conceptual Classes from Domain
  • Focus on real-world entities (Customer, Bank, Account)

  • Analyze use case descriptions (Withdraw Cash ‚Üí Account, Transaction)

  • Identify common behavior (Account ‚Üí SavingsAccount, CurrentAccount)

  • 7) Define Class Responsibilities
  • Assign clear responsibilities, follow Single Responsibility Principle
  • 8) Identify Attributes
  • Attributes should represent state (Account ‚Üí accountNumber, balance)
  • 9) Identify Operations (Methods)
  • Operations represent behavior (debit(), credit(), validatePIN())
  • 10) Define Relationships
  • Association, Aggregation/Composition, Inheritance, Dependency
  • 11) Ensure High Cohesion
  • Class should focus on single purpose, avoid unrelated methods
  • 12) Reduce Coupling
  • Minimize dependencies between classes, use interfaces
  • 13) Validate Using CRC Cards
  • Class‚ÄìResponsibility‚ÄìCollaborator helps refine responsibilities
  • 14) Refine Through Iteration
  • Review and adjust, remove redundancy, simplify hierarchy
Q: Importance of Proper Class Identification
  • Accurate system representation
  • Easier implementation
  • Better reuse
  • Reduced maintenance effort
Refactoring and Reification
Q: What is Refactoring?
  • Definition: Process of improving internal structure of classes without changing their external behavior
  • Purpose: Improve readability, reduce complexity, remove code duplication, improve maintainability
  • Refactoring Activities:
  • ‚Ä¢ Splitting large classes
  • ‚Ä¢ Renaming classes, attributes, methods
  • ‚Ä¢ Moving methods to appropriate classes
  • ‚Ä¢ Simplifying inheritance hierarchies
  • ‚Ä¢ Replacing inheritance with composition
Q: Refactoring Example
  • Before: Class Transaction with validate(), log(), notifyUser()
  • After: Transaction, Logger, NotificationService (responsibilities separated)
Q: What is Reification?
  • Definition: Process of converting an abstract concept, behavior, or relationship into an explicit class
  • Purpose: Make design more flexible, enable dynamic behavior, support reuse and extension
  • When Reification is Applied:
  • ‚Ä¢ When behavior varies at runtime
  • ‚Ä¢ When relationships need independent control
  • ‚Ä¢ When rules or policies change frequently
Q: Reification Example
  • Without Reification: if(paymentType == CARD) / if(paymentType == CASH)
  • With Reification: Payment class with CardPayment and CashPayment subclasses
Q: Refactoring vs Reification Comparison
Aspect Refactoring Reification
Meaning Improves structure Converts concept into class
Behavior Change No No
Focus Code/design cleanup Explicit representation
Result Cleaner design More flexible design
Q: Common Refactoring Techniques
  • Extract Class: Breaking large class into smaller, responsibility-focused classes
  • Rename Method: Use meaningful names
  • Move Method: Move to appropriate class
  • Remove unused attributes: Clean up unused code
  • Simplify inheritance hierarchy: Reduce complexity

Package Dependencies
Q: What is Package Dependency?
  • Definition: Relationship that shows one package depends on another package for its functionality
  • Purpose: Organize related classes and model dependencies between packages
Q: Types of Dependencies in Package
Dependency Type Meaning Strength Example
Usage (<>) Uses services/elements of another package Medium UI package uses Business Logic
Import (<>) Full public access, can reference directly Strong Package A imports Package B classes
Access (<>) Qualified access using package name Weak Package A accesses Package B with qualified names
Merge (<>) Combines contents of packages Strong Application merges framework package
Realization Implements specifications/interfaces Medium Implementation realizes Interface package
Q: Detailed Explanation of Dependencies
  • 1) Usage Dependency (<>):
  • Client package depends on supplier package, supplier provides services
  • Representation: Dashed arrow from using to used package
  • 2) Import Dependency (<>):
  • Imported elements can be referenced directly without package prefix
  • Creates tight coupling
  • 3) Access Dependency (<>):
  • Elements must be referenced using package name (weaker than import)
  • 4) Merge Dependency (<>):
  • Used to extend or refine package definitions, common in framework extension
  • Elements of merged package become part of target
  • 5) Realization Dependency:
  • One package provides implementation, another provides interfaces/abstract definitions
  • Representation: Dashed line with hollow arrow
Q: Importance of Package Dependencies
  • Helps manage large systems
  • Controls coupling between packages
  • Improves modularity
  • Aids impact analysis during changes
