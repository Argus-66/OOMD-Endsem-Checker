Class Design Overview
Q: Overview of Class Design
  • Class Design: Refining analysis classes into implementable code structures
  • Steps:
  • 1. Identify Classes: From use cases & domain model
  • 2. Define Attributes: Data each class holds
  • 3. Define Methods: Operations/behaviors
  • 4. Define Relationships: Inheritance, associations
  • 5. Apply Design Principles: SOLID principles
  • 6. Optimize: Performance, memory, maintainability
Realizing Use Cases
Q: Realizing Use Cases with Classes
  • Realization: Converting use case scenarios into class interactions
  • Process:
  • 1. Identify Classes: Actors ‚Üí boundary, entities ‚Üí entity, control logic ‚Üí control classes
  • 2. Assign Responsibilities: Determine which class handles each step
  • 3. Define Collaborations: How classes communicate (sequence diagrams)
  • 4. Refine Interactions: Add parameters, return types, error handling
Inheritance Adjustments
Q: Adjustment of Inheritance
  • Goal: Optimize inheritance hierarchies for better design
  • Techniques:
  • 1. Eliminate Unnecessary Inheritance: Remove if "is-a" relationship doesn't exist
  • 2. Multiple Inheritance ‚Üí Interfaces: Use interfaces to avoid diamond problem
  • 3. Deep Hierarchies ‚Üí Flat: Reduce complexity by flattening
  • 4. Favor Composition over Inheritance: "has-a" instead of "is-a" when flexible
Q: Inheritance vs Composition
Aspect Inheritance (is-a) Composition (has-a)
Relationship Child "is a" type of Parent Object "has a" component
Coupling Tight coupling Loose coupling
Flexibility Less flexible (compile-time) More flexible (runtime)
Example Dog extends Animal Car has Engine
When to Use True specialization needed Reuse functionality without inheritance
Associations to Attributes
Q: Associations to Attributes
  • Converting class diagram associations into code attributes
  • Rules:
  • 1. One-to-One: Single reference attribute (Person has License)
  • 2. One-to-Many: Collection attribute List/Array (Teacher has Students[])
  • 3. Many-to-Many: Collections on both sides (Student[] has Course[], Course[] has Student[])
  • 4. Bidirectional: Both classes hold references to each other
  • 5. Unidirectional: Only one class has reference

Adding Operations
Q: Criteria for Adding Operations
  • 1. Information Expert: Assign operation to class with most information
  • 2. Single Responsibility: Operation matches class purpose
  • 3. Encapsulation: Keep data and operations together
  • 4. Cohesion: Operation logically belongs to class
  • Example: calculateSalary() goes in Employee class, not in Company class
Q: Types of Operations
Type Purpose Example
Constructor Initialize object new Employee()
Accessor (Getter) Read attribute value getName()
Mutator (Setter) Modify attribute value setName()
Manager Create/destroy related objects addEmployee()
Iterator Traverse collection getNextEmployee()

Q: Reusing Methods
  • 1. Inheritance: Child class reuses parent methods
  • 2. Composition: Delegate to component object's methods
  • 3. Utility Classes: Static helper methods (Math.sqrt())
  • 4. Interfaces: Different classes implement same method signature
  • Benefits: Less code duplication, easier maintenance
Refactoring
Q: What is Refactoring?
  • Refactoring: Restructuring code to improve design without changing behavior
  • Goals: Better readability, maintainability, performance
  • When to Refactor:
  • ‚Ä¢ Code smells (duplicates, long methods, large classes)
  • ‚Ä¢ Before adding new features
  • ‚Ä¢ During code reviews
  • Rule: Always maintain tests to ensure behavior unchanged
Q: Common Refactoring Techniques
Technique Problem Solution
Extract Method Long, complex method Break into smaller methods
Rename Unclear names Use meaningful names
Move Method Method in wrong class Move to appropriate class
Replace Magic Numbers Hardcoded values Use named constants
Extract Class Class doing too much Split into focused classes
OCL Constraints
Q: Object Constraint Language (OCL)
  • OCL: Formal language to specify constraints on UML models
  • Used For:
  • 1. Invariants: Always true conditions (age >= 0)
  • 2. Pre-conditions: Must be true before operation (balance > amount)
  • 3. Post-conditions: Must be true after operation (balance reduced)
  • Benefits: Precise, unambiguous specifications
