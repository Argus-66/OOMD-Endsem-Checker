Design Pattern Basics
Q: What is a Design Pattern?
  • Design Pattern: Reusable solution template for common design problems
  • Benefits:
  • ‚Ä¢ Proven solutions to recurring problems
  • ‚Ä¢ Common vocabulary for developers
  • ‚Ä¢ Speeds up development
  • ‚Ä¢ Improves code maintainability
  • Not: Finished code you can copy-paste, but a template to adapt
Q: Elements of a Design Pattern
Element Description
Pattern Name Identifies the pattern (e.g., Singleton, Observer)

Observer Pattern
Q: Observer Pattern Explained
  • Purpose: Define one-to-many dependency where state change in one object notifies all dependents
  • Components:
  • 1. Subject: Maintains list of observers, notifies on state change
  • 2. Observer: Interface with update() method
  • 3. Concrete Observer: Implements update() to react to changes
  • Use Case: Event systems, MVC (Model notifies Views), Newsletter subscriptions
  • Benefits: Loose coupling, dynamic relationships
Subject: attach(observer), detach(observer), notify()
Observer: update()
When Subject changes ‚Üí calls notify() ‚Üí all observers' update() called
Strategy Pattern
Q: Strategy Pattern Explained
  • Purpose: Define family of algorithms, encapsulate each, make them interchangeable
  • Components:
  • 1. Context: Uses a Strategy interface
  • 2. Strategy Interface: Common interface for all algorithms
  • 3. Concrete Strategies: Different implementations of algorithm
  • Use Case: Payment methods (CreditCard, PayPal), Sorting algorithms, Compression formats
  • Benefits: Swap algorithms at runtime, avoid conditionals, easier to add new strategies
Context ‚Üí Strategy Interface ‚Üê Concrete Strategies
Example: PaymentContext uses IPaymentStrategy
Strategies: CreditCardPayment, PayPalPayment, CryptoPayment
Adapter Pattern
Q: Adapter Pattern Explained
  • Purpose: Convert interface of a class into another interface clients expect
  • Components:
  • 1. Target: Interface expected by client
  • 2. Adaptee: Existing class with incompatible interface
  • 3. Adapter: Wraps Adaptee and implements Target interface
  • Use Case: Legacy system integration, Third-party library compatibility
  • Benefits: Reuse existing code without modification
Client ‚Üí Target Interface ‚Üê Adapter ‚Üí Adaptee
Example: USB-C Adapter for USB-A devices
Factory Pattern
Q: Factory Pattern Explained
  • Purpose: Create objects without specifying exact class
  • Types:
  • 1. Simple Factory: Static method returns objects based on parameter
  • 2. Factory Method: Subclasses decide which class to instantiate
  • 3. Abstract Factory: Create families of related objects
  • Use Case: Document creators (PDF, Word), UI themes (Windows, Mac), Database connections
  • Benefits: Loose coupling, easier to extend with new types
Factory.createProduct(type) ‚Üí returns Product
Example: ShapeFactory.create("circle") ‚Üí Circle object
Singleton Pattern
Q: Singleton Pattern Explained
  • Purpose: Ensure a class has only ONE instance and provide global access
  • Implementation:
  • 1. Private constructor (prevent external instantiation)
  • 2. Private static instance variable
  • 3. Public static getInstance() method
  • Use Case: Logger, Configuration manager, Database connection, Thread pool
  • Benefits: Controlled access, saves resources
  • Drawbacks: Hard to unit test, can violate Single Responsibility
private static instance
private Constructor() { }
public static getInstance() {
if(instance == null) instance = new Singleton();
return instance;
}
Composite Pattern
Q: Composite Pattern Explained
  • Purpose: Compose objects into tree structures to represent part-whole hierarchies
  • Components:
  • 1. Component: Interface for all objects in composition
  • 2. Leaf: Individual objects (no children)
  • 3. Composite: Container for Components (has children)
  • Use Case: File systems (Folder contains Files/Folders), UI components (Panel contains Buttons), Graphics (Group contains Shapes)
  • Benefits: Treat individual and composite objects uniformly

Decorator Pattern
Q: Decorator Pattern Explained
  • Purpose: Add responsibilities to objects dynamically without altering structure
  • Components:
  • 1. Component: Interface for objects
  • 2. Concrete Component: Base object
  • 3. Decorator: Wraps component, adds behavior
  • Use Case: Pizza toppings, Text formatting (Bold, Italic), Java I/O streams
  • Benefits: More flexible than inheritance, add/remove at runtime
BasePizza ‚Üí ToppingDecorator1 ‚Üí ToppingDecorator2
Example: SimplePizza + CheeseDecorator + PepperoniDecorator
Each decorator adds cost and description
Command Pattern
Q: Command Pattern Explained
  • Purpose: Encapsulate a request as an object, allowing parameterization and queuing
  • Components:
  • 1. Command: Interface with execute() method
  • 2. Concrete Command: Implements execute(), binds action to receiver
  • 3. Receiver: Object that performs actual work
  • 4. Invoker: Triggers command execution
  • Use Case: Undo/Redo, Button actions, Macro recording, Transaction management
  • Benefits: Decouple sender and receiver, support undo, queue operations
Invoker ‚Üí Command.execute() ‚Üí Receiver.action()
Example: RemoteControl ‚Üí LightOnCommand ‚Üí Light.turnOn()
Template Method Pattern
Q: Template Method Pattern Explained
  • Purpose: Define skeleton of algorithm in base class, let subclasses override specific steps
  • Components:
  • 1. Abstract Class: Defines template method and abstract operations
  • 2. Concrete Class: Implements abstract operations
  • Use Case: Game AI (common flow, different behaviors), Data parsing, Report generation
  • Benefits: Code reuse, control over algorithm structure
AbstractClass {
templateMethod() { step1(); step2(); step3(); }
abstract step2();
}
ConcreteClass { step2() { /* specific implementation */ } }
Pattern Comparisons
Q: Strategy vs Template Method
Aspect Strategy Template Method
Mechanism Composition (has-a) Inheritance (is-a)
Flexibility Change at runtime Fixed at compile-time
Variation Point Entire algorithm Specific steps
Use When Multiple interchangeable algorithms Common structure, varying details
Q: Adapter vs Decorator
Aspect Adapter Decorator
Purpose Convert interface to match another Add responsibilities without changing interface
Interface Change Changes interface Keeps same interface
Intent Make incompatible interfaces work Enhance functionality
Example USB-C to USB-A adapter Coffee + Milk decorator

Aspect Factory Method Abstract Factory
Scope Creates ONE product Creates FAMILIES of related products
Implementation Single method in subclass Multiple factory methods

SOLID Principles
Q: SOLID Principles Overview
Principle Description Benefit
S - Single Responsibility Class should have one reason to change Easier maintenance
O - Open/Closed Open for extension, closed for modification Add features without breaking existing code
L - Liskov Substitution Subclasses should be substitutable for base class Proper inheritance
I - Interface Segregation Many specific interfaces better than one general Clients not forced to implement unused methods
D - Dependency Inversion Depend on abstractions, not concretions Loose coupling, easier testing
Anti-Patterns
Q: Common Anti-Patterns (Bad Practices)
  • Anti-Pattern: Common bad solution to recurring problem
  • Examples:
  • 1. God Object: One class knows/does too much
  • 2. Spaghetti Code: Complex, tangled control flow
  • 3. Golden Hammer: Using same solution for every problem
  • 4. Copy-Paste Programming: Duplicating code instead of reusing
  • 5. Magic Numbers: Hardcoded values without explanation
