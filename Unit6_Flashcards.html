Design Pattern Basics
Q: What is Design Pattern?
  • Design Pattern: General reusable solution to commonly occurring problem in software design within given context
  • Key Characteristics:
  • ‚Ä¢ Not complete design or code
  • ‚Ä¢ Language independent
  • ‚Ä¢ Improves reusability and maintainability
  • ‚Ä¢ Based on best practices
  • Purpose: Solve recurring design problems, improve communication, promote good OO design, reduce complexity
  • Examples: Singleton, Factory, Observer
Q: Entities/Elements of Design Pattern
Entity/Element Description Purpose
Pattern Name Identifies the design pattern uniquely Used for communication among designers
Intent Explains purpose of pattern Describes what pattern does
Problem Describes design problem addressed Explains when pattern should be used
Context Defines situation where problem occurs Explains conditions before applying pattern
Solution Describes structure of classes and objects Explains relationships and responsibilities
Participants Classes/objects that take part in pattern Each has defined role (e.g., Subject, Observer)
Consequences Results of applying pattern Includes benefits and trade-offs
Implementation Guidelines to implement pattern Language-specific issues
  • Three Most Important Parts:
  • 1. Context: WHERE problem happens
  • 2. Problem: WHAT the issue is
  • 3. Solution: HOW issue is solved
Q: Types/Categories of Design Patterns
Category Focus Purpose Examples

  • Summary: Creational ‚Üí How objects made | Structural ‚Üí How objects connected | Behavioral ‚Üí How objects communicate
Observer Pattern
Q: Explain Observer Design Pattern
  • Observer Pattern: Defines one-to-many relationship where when one object (subject) changes state, all dependent objects (observers) are notified and updated automatically
  • Intent: Maintain consistency between related objects, avoid tight coupling, support dynamic addition/removal of observers
  • Participants:
  • 1. Subject: Maintains list of observers, provides attach/detach methods, notifies observers when state changes
  • 2. Observer: Defines update interface, receives notification from subject
  • 3. ConcreteSubject: Implements Subject, stores state, triggers notification
  • 4. ConcreteObserver: Implements Observer, updates state based on subject
  • Working: Observers register ‚Üí Subject changes state ‚Üí Subject calls notify() ‚Üí All observers receive update() ‚Üí Observers update themselves
  • Example: YouTube (Channel = Subject, Subscribers = Observers); Stock market notifications; ATM account balance change
  • Advantages: Loose coupling, supports Open-Closed Principle, dynamic subscription, promotes scalability
  • Disadvantages: Unexpected updates, performance overhead with many observers, notification order not guaranteed
Strategy Pattern
Q: Explain Strategy Design Pattern
  • Strategy Pattern: Defines family of algorithms, encapsulates each one, makes them interchangeable, allowing algorithm to vary independently from client
  • Intent: Avoid multiple conditionals, select behavior dynamically at runtime, promote loose coupling between client and algorithm
  • Participants:
  • 1. Strategy (Interface): Declares common interface for all algorithms (e.g., PaymentStrategy)
  • 2. Concrete Strategy: Implements algorithm (e.g., CreditCardPayment, UPIPayment, CashPayment)
  • 3. Context: Uses Strategy reference, delegates execution (e.g., PaymentProcessor)
  • Working: Context holds Strategy ‚Üí Client selects Concrete Strategy ‚Üí Strategy passed to Context ‚Üí Context calls strategy's algorithm ‚Üí Algorithm executes
  • Example: Payment system (multiple payment methods), Travel modes (Walk, Bike, Car), Sorting algorithms
  • Advantages: Eliminates conditional logic, supports Open-Closed Principle, easy to add new strategies, improves maintainability
  • Disadvantages: Increases number of classes, client must be aware of strategies, object creation overhead
  • Use When: Multiple algorithms for same task, behavior must change at runtime, conditionals become complex
State Pattern
Q: Explain State Design Pattern
  • State Pattern: Allows object to alter its behavior when internal state changes, appearing as if object changed its class
  • Intent: Encapsulate state-specific behavior, eliminate complex conditional logic, make state transitions explicit
  • Participants:
  • 1. Context: Maintains reference to current state, delegates behavior to current state object
  • 2. State (Interface): Declares interface for state-specific behavior
  • 3. Concrete State: Implements behavior for particular state, handles state transitions
  • Example: ATM Machine (States: Idle, CardInserted, Authenticated, Dispensing); Vending machine states; TCP connection states
  • Advantages: Removes conditional statements, improves maintainability, supports Open-Closed Principle
  • Disadvantages: Increased number of classes, can be overkill for simple state logic
Adapter Pattern
Q: Discuss Adapter Design Pattern
  • Adapter Pattern: Converts interface of a class into another interface that clients expect
  • Intent: Enable compatibility between incompatible interfaces, promote reuse of existing classes, avoid changing legacy code
  • Types:
  • 1. Class Adapter: Uses inheritance, adapter inherits from adaptee
  • 2. Object Adapter: Uses composition, adapter holds reference to adaptee (Preferred)
  • Components: Client ‚Üí Target Interface ‚Üí Adapter ‚Üí Adaptee
  • Example: Power plug adapter, Adapter between old payment gateway and new API, Media player supporting new formats
  • Advantages: Promotes reusability, improves flexibility, does not affect existing code
  • Disadvantages: Increases number of classes, adds extra level of indirection
Communication Patterns
Q: What is Communication Pattern?
  • Communication Pattern: Reusable design solution specifying how objects communicate to achieve particular behavior
  • Key Points: Focuses on interaction between objects, defines message flow and responsibilities, improves maintainability, avoids tight coupling
  • Examples: Observer Pattern, Client-Server Pattern, Publish-Subscribe Pattern, Request-Reply Pattern
Q: Publisher-Subscriber Design Pattern
  • Pub-Sub Pattern: Publishers generate events/messages, subscribers receive them, without publisher knowing subscribers directly
  • Structure - Participants:
  • 1. Publisher: Generates events, doesn't know subscribers, publishes to channel (e.g., Stock price updater)
  • 2. Subscriber: Registers interest in events, receives updates, can subscribe/unsubscribe dynamically (e.g., Display, alerts)
  • 3. Event Channel/Broker: Intermediary, receives from publisher, distributes to subscribers (e.g., Message queue, event bus)
  • Working: Subscribers register ‚Üí Publisher publishes event ‚Üí Event channel forwards to all subscribers ‚Üí Subscribers process event
  • Variants:
  • 1. Topic-Based: Messages under specific topics, subscribers receive only subscribed topics (e.g., News: Sports, Politics)
  • 2. Content-Based: Messages filtered by content, subscribers define conditions (e.g., Notify if price > ‚Çπ1000)
  • 3. Broker-Based: Central broker handles routing, full decoupling (e.g., Kafka, RabbitMQ)
  • 4. Event-Based: Triggered by events, asynchronous (e.g., GUI events, sensors)
  • Advantages: Loose coupling, high scalability, easy addition/removal of subscribers, asynchronous communication
  • Disadvantages: Difficult debugging, message delivery order issues, broker can become bottleneck
Q: Client-Dispatcher-Server Pattern
  • Client-Dispatcher-Server: Client sends request ‚Üí Dispatcher receives and routes ‚Üí Server processes and returns response
  • Need: Multiple servers provide different services, clients shouldn't know server details, centralized request handling
  • Components:
  • 1. Client: Initiates request, doesn't know which server handles it, receives response (e.g., ATM, web browser, mobile app)
  • 2. Dispatcher: Intermediate controller, routes to appropriate server, contains routing logic (forwards request, returns response)
  • 3. Server: Performs actual processing, provides specific services (e.g., Authentication Server, Payment Server, Order Server)
  • Working: Client sends to Dispatcher ‚Üí Dispatcher analyzes request ‚Üí Selects Server ‚Üí Server processes ‚Üí Response to Dispatcher ‚Üí Dispatcher forwards to Client
  • Example (ATM): Client = ATM Machine, Dispatcher = Transaction Controller, Servers = Authentication, Account services
  • Advantages: Loose coupling, centralized routing, easy to add servers, improved scalability, better maintainability
  • Disadvantages: Dispatcher can be bottleneck, single point of failure, slight overhead
Q: View Handler Pattern
  • View Handler: Central controller (view handler) responsible for creating, managing, switching, and coordinating views in application
  • Need: Multiple screens, complex navigation, avoid direct view-to-view communication (reduces coupling)
  • Components:
  • 1. View: User interface, displays data, captures input, minimal logic, doesn't decide next view (e.g., Login, Dashboard)
  • 2. View Handler: Central controller, manages views, controls navigation, creates/destroys views, maintains lifecycle (CORE component)
  • 3. Model: Business data and logic, shared among views
  • Working: User acts on view ‚Üí View sends request to Handler ‚Üí Handler decides next view ‚Üí Handler loads view ‚Üí Data fetched ‚Üí New view displayed
  • Example (ATM): Screens: Welcome, PIN Entry, Transaction Menu, Receipt | View Handler manages flow between screens
  • Advantages: Loose coupling, centralized navigation logic, easy to add views, improves maintainability, cleaner UI design
  • Disadvantages: View Handler can become complex, single point of control
Advanced Pattern Concepts
Q: Counted Pointer Example
  • Counted Pointer: Pointer (reference) maintaining count of how many objects refer to same resource; deallocated only when reference count = 0
  • Purpose: Manage shared resources safely, avoid premature deletion, prevent memory leaks, support controlled object lifetime
  • Working Principle:
  • ‚Ä¢ Object acquires resource ‚Üí reference count incremented
  • ‚Ä¢ Object releases resource ‚Üí reference count decremented
  • ‚Ä¢ Reference count = 0 ‚Üí resource destroyed
  • Example (Shared Printer):
  • Step 1: Printer created ‚Üí Ref Count = 1
  • Step 2: Another object uses printer ‚Üí Ref Count = 2
  • Step 3: One object releases ‚Üí Ref Count = 1
  • Step 4: Last object releases ‚Üí Ref Count = 0 ‚Üí Printer deleted
  • Used In: Database connection pools, file handling, shared configuration objects, smart pointers (C++ shared_ptr)
  • Advantages: Automatic memory management, prevents dangling pointers, efficient resource sharing, simple concept
  • Disadvantages: Overhead of maintaining count, circular references cause memory leaks, not suitable for all scenarios
  • Analogy: Library book system - book removed only when no one has borrowed it

Q: Define Idioms. What Can Idioms Provide? Where to Find?
  • Idiom: Low-level, language-specific solution to common programming problem
  • What Idioms Provide:
  • 1. Efficient Coding Practices: Best ways to use language features
  • 2. Improved Readability: Standard coding style
  • 3. Performance Optimization: Efficient memory and resource handling
  • 4. Safe Programming: Avoid common programming errors
  • Examples: C++ RAII (Resource Acquisition Is Initialization), Java try-with-resources, Python context managers
  • Where to Find Idioms:
  • ‚Ä¢ Programming language documentation
  • ‚Ä¢ Design pattern and idiom books
  • ‚Ä¢ Expert codebases
  • ‚Ä¢ Language-specific guidelines
  • ‚Ä¢ Open-source projects
